## Key decisions made
- Decision: Hardcode Airtable base/table IDs (including the new Orders table) as constants in the function files rather than env vars.
  Why: Matches the existing convention already used for `BASE_ID` / `TABLE_ID`; IDs are not secret and rarely change.
- Decision: Order-recording is best-effort — a failed Airtable write is logged but does NOT fail the customer response.
  Why: Stock has already been committed by the time we write the order; telling the buyer their paid-for order failed would be wrong. Logs surface the failure for manual reconciliation.
- Decision: Server is the source of truth for the discount amount; client `DISCOUNT_CODES` is UX-only.
  Why: Prevents tampering — confirmed in testing that a fake code with a tampered client total is rejected and the server-computed total is recorded.
- Decision: Discount applies to subtotal; delivery unchanged; no minimum spend; reusable shared code.
  Why: User's choice given the printed-card design (every card shows the same code). Standard, merchant-safe math.
- Decision: Notifications via Airtable email automation + Gmail push on the phone, not Airtable native push.
  Why: Airtable doesn't have a reliable native-push automation action; email → Gmail push is the dependable real-time path.
